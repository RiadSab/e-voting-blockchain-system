# E-Voting Platform — Full Architecture Document

---

## 1. Overview

A government-grade, trustless, blockchain-based e-voting platform built on Ethereum.
The core principle is that no single entity — including the platform operator — can
unilaterally influence, read, or tamper with any vote or election outcome.

Privacy, integrity, and verifiability are enforced cryptographically, not by trust.

## 3. Roles and Actors

### Platform Super Admins
- Control the platform-level Gnosis Safe
- Responsible for creating elections and assigning election authorities
- 2 of 3 must approve any platform-level action
- Have no control over individual election operations after creation

### Election Authority
- A separate Gnosis Safe per election
- Assigned during election creation by platform super admins
- Responsible for voter registration, opening and closing the election
- Has no access to ballot contents — privacy is cryptographically enforced
- Typically a government ministry or electoral commission with their own signers

### Guardians
- Purely cryptographic role — no administrative power
- Hold secret key shares used for threshold decryption of the final tally
- Selected per election from a pre-approved registry
- Participate in the key ceremony (submit public key share)
- Participate in tally decryption (submit partial decryption + ZK proof)
- Secret shares never touch the platform — generated and stored by guardians themselves

### Voters
- Registered via identity commitments (Semaphore)
- Never need ETH or a crypto wallet for the happy path
- Generate and encrypt their ballot entirely client-side
- Submit vote to backend which relays it on-chain

## 4. Key Concepts

### Gnosis Safe (Multisig)
Every administrative action goes through a Gnosis Safe smart contract wallet.
A threshold of signers (e.g. 2 of 3) must approve before any transaction executes.
The blockchain enforces this — the platform backend cannot bypass it.

### ElGamal Threshold Encryption
The election public key is jointly generated by guardians during a key ceremony.
No single guardian holds the full private key — only shares.
Ballots are encrypted with the joint public key.
Decryption requires k of n guardians to cooperate at tally time.

### Vector Ballot Encoding
A vote is not an encrypted index. It is a vector of encrypted bits — one per candidate.
The chosen candidate encrypts to 1, all others encrypt to 0.
This prevents brute-force ciphertext comparison attacks.

```
Candidates: [ Alice, Bob, Carol ]
Voter chooses Bob:
Ballot = [ enc(0), enc(1), enc(0) ]
```

### Homomorphic Tally
ElGamal is multiplicatively homomorphic:
  enc(a) * enc(b) = enc(a + b)

Encrypted ballots are aggregated directly on-chain without decryption:

```
Voter 1: [ enc(0), enc(1), enc(0) ]
Voter 2: [ enc(1), enc(0), enc(0) ]
Voter 3: [ enc(0), enc(1), enc(0) ]
─────────────────────────────────
Tally:   [ enc(1), enc(2), enc(0) ]
           Alice=1  Bob=2  Carol=0
```

### Semaphore Identity
Each voter generates a Semaphore identity (private key + identity commitment) client-side.
The identity commitment is registered on-chain.
When voting, the voter proves membership in the registered set using a ZK proof
without revealing which commitment belongs to them.
A nullifier prevents double voting without linking votes to identities.

### ZK Proofs Used

| Proof | Purpose |
|---|---|
| Semaphore membership proof | Proves voter is registered without revealing identity |
| Ballot validity proof (disjunctive Chaum-Pedersen) | Proves each position encrypts 0 or 1 and exactly one is 1 |
| Guardian key share proof | Proves guardian knows their secret share |
| Partial decryption proof (Chaum-Pedersen) | Proves partial decryption was computed correctly |

### Meta-Transactions (ERC-2771)
Voters sign their vote data client-side.
The backend relays the signed data and pays gas.
The contract recovers and verifies the original signer.
The relayer cannot forge or modify votes.
If the relayer censors a vote, the voter has a signed receipt as proof.
The voter can also submit directly if they have ETH — the contract accepts both paths.

### IPFS Content Addressing
Large data is stored on IPFS. Only the content hash (CID) goes on-chain.
Anyone can fetch the data from IPFS and verify it matches the on-chain hash.
The platform runs its own IPFS node — no dependency on external pinning services.

## 5. Smart Contract Structure

```
ElectionFactory
  └── Election (one deployed per election)
        ├── election public key (ElGamal joint key from ceremony)
        ├── guardian registry (addresses + public shares)
        ├── voter registry (Semaphore identity commitments)
        ├── ballot box (encrypted ballots)
        ├── homomorphic tally (running encrypted aggregate per candidate)
        ├── partial decryptions (submitted by guardians at tally time)
        └── final result (published after threshold decryptions received)
```

### Access Control

```
ElectionFactory.createElection()       → only Platform Safe
Election.registerVoters()              → only Election Authority Safe
Election.openElection()                → only Election Authority Safe
Election.closeElection()               → only Election Authority Safe
Election.castVote()                    → open (verified by ZK proofs on-chain)
Election.submitPartialDecryption()     → only registered guardians
Election.finalizeResult()              → automatic after threshold met
```

### What Goes Where

| Data                       | On-chain | IPFS        | DB      |
|----------------------------|----------|-------------|---------|
| Election metadata CID      | ✓        | ✓ (content) | cached  |
| Election public key        | ✓        | —           | —       |
| Guardian addresses         | ✓        | —           | —       |
| Guardian public key shares | —        | —           | ✓       |
| Threshold k                | ✓        | —           | —       |
| Identity commitments       | ✓        | —           | —       |
| Nullifiers                 | ✓        | —           | —       |
| Encrypted ballots          | ✓        | —           | —       |
| Homomorphic tally          | ✓        | —           | —       |
| Partial decryptions        | ✓        | —           | ✓       |
| ZK proofs (tally)          | ✓        | ✓ (audit)   | —       |
| Final result               | ✓        | ✓ (audit)   | —       |
| Guardian secret shares     | ✗ never  | ✗ never     | ✗ never |
| Plaintext votes            | ✗ never  | ✗ never     | ✗ never |

## 6. Business Flows

---

### 6.1 Guardian Registration (Pre-Election)

A super admin registers an institution as a guardian in the platform registry.
This builds a pool of available guardians before any election is created.

```
Super admin submits:
  { name, institution, email, ethereumAddress }
        │
        ▼
Spring Boot adds guardian to registry (DB)
Status: PENDING
        │
        ▼
Platform sends invitation email to guardian
        │
        ▼
Guardian visits platform → authenticates via wallet signature (see 6.2)
        │
        ▼
Guardian account activated
Status: ACTIVE
```

---

### 6.2 Wallet-Based Authentication (All Privileged Actors)

Applies to super admins, election authority signers, guardians, and voters.
No passwords — identity proven by controlling an Ethereum private key.

```
Actor requests login with their Ethereum address
        │
        ▼
Spring Boot generates random nonce:
  "Authenticate to E-Voting Platform: <random_hex>"
  Stored with short expiry in Redis
        │
        ▼
Frontend presents nonce to actor's wallet (MetaMask, hardware wallet)
Actor signs → signature sent to Spring Boot
        │
        ▼
Spring Boot:
  recoveredAddress = ecrecover(nonce, signature)
  check: recoveredAddress == registered address in DB
  check: nonce not expired
        │
        ▼
Match   → JWT issued with role (SUPER_ADMIN | ELECTION_AUTHORITY | GUARDIAN | VOTER)
No match → 401 rejected
```

### 6.3 Election Creation and Approval

```
Admin 1 (super admin) creates election draft:
  {
    title,
    description,
    candidates[],
    guardians[],                    ← selected from guardian registry
    threshold,                      ← e.g. 3 of 5
    startDate,
    endDate,
    electionAuthoritySafeAddress    ← pre-deployed Safe of election authority
  }
        │
        ▼
Spring Boot:
  computes draftHash = keccak256(all draft fields)
  stores draft in DB
  Status: DRAFT
        │
        ▼
Admin 1 signs draftHash with wallet → approval stored:
  { adminId, signature, timestamp }
        │
        ▼
Platform notifies Admin 2 and Admin 3
        │
        ▼
Admin 2 logs in → reviews draft → signs draftHash
  approval stored
  Status: 1_OF_2_APPROVED
        │
        ▼
Admin 3 logs in → reviews draft → signs draftHash
  approval stored
  Status: APPROVED

Note: Admin 1 (creator) cannot be one of the two approvers.
      If draft is modified, all existing signatures are invalidated.
        │
        ▼
Trigger: key ceremony initiated (see 6.4)
```

### 6.4 Key Ceremony

Runs after election draft is approved.
All selected guardians must participate before the election can be deployed.

```
Platform notifies all selected guardians:
  "You are guardian for election X. Submit your key share by <deadline>."
        │
        ▼
Each guardian independently:
        │
        ▼
Guardian logs in (wallet auth)
Browser loads Rust WASM crypto module
Browser generates (entirely client-side):
  - secret share si          ← never leaves the browser
  - public share Pi = g^si   ← submitted to platform
  - ZK proof of knowledge of si
        │
        ▼
Guardian downloads encrypted secret share file:
  {
    electionId,
    guardianIndex,
    secretShare: encrypted with AES-256-GCM using guardian's wallet key,
    hint: "decrypt with your Ethereum private key"
  }
Guardian stores this file offline (hardware key, secure vault, notary)
        │
        ▼
Guardian submits to platform:
  { publicShare: Pi, zkProof }
        │
        ▼
Spring Boot verifies ZK proof
Stores { guardianId, publicShare, zkProof } in DB
        │
        ▼
Once ALL n guardians have submitted:
  Rust sidecar combines public shares:
    joint public key K = P1 * P2 * P3 * P4 * P5
        │
        ▼
Election metadata + joint public key K → uploaded to IPFS
IPFS returns metadataCID
        │
        ▼
Spring Boot instructs TS TX Service:
  Platform Safe proposes transaction to Safe Transaction Service:
    ElectionFactory.createElection(
      metadataCID,
      electionPublicKey K,
      guardianAddresses[],
      threshold,
      startDate,
      endDate,
      electionAuthoritySafeAddress
    )
        │
        ▼
Platform Admin 2 and Admin 3 confirm via Safe Transaction Service
Safe executes → Election contract deployed on-chain
        │
        ▼
TS Indexer detects ElectionCreated event
Spring Boot updates DB
Status: PENDING
```

### 6.7 Vote Casting

```
Voter visits platform (authenticated)
        │
        ▼
Browser fetches from TS Indexer:
  - current Merkle root
  - Merkle proof for voter's identity commitment
        │
        ▼
Browser fetches from chain (or indexer cache):
  - election public key K
        │
        ▼
Voter selects candidate — choice never leaves the browser
        │
        ▼
Browser loads Rust WASM crypto module and:

  Step 1 — Encrypts ballot as vector:
    ballot = [
      ElGamal.encrypt(0, K),   // candidate 0 — not chosen
      ElGamal.encrypt(1, K),   // candidate 1 — chosen
      ElGamal.encrypt(0, K),   // candidate 2 — not chosen
    ]

  Step 2 — Generates ballot validity proof (disjunctive Chaum-Pedersen):
    proves: each position encrypts 0 or 1
    proves: exactly one position encrypts 1
    without revealing: which position is 1

  Step 3 — Generates Semaphore ZK proof:
    proves: voter's identity commitment is in the Merkle tree
    generates: nullifier (unique per voter per election)
    without revealing: which commitment belongs to this voter
        │
        ▼
Voter sends to Spring Boot:
  {
    encryptedBallot: [ enc(0), enc(1), enc(0) ],
    ballotValidityProof,
    semaphoreProof,
    nullifier,
    merkleRoot
  }
  Plaintext vote choice never leaves the browser.
        │
        ▼
Spring Boot pre-verifies (before spending gas):
  - Merkle root matches a valid snapshot
  - Nullifier not already recorded on-chain
  - ZK proof structurally valid
  Rejects invalid submissions early
        │
        ▼
Spring Boot sends to TS TX Service
TS TX Service submits meta-transaction:
  Election.castVote(encryptedBallot, ballotValidityProof, semaphoreProof, nullifier)
        │
        ▼
Election contract verifies on-chain:
  - Semaphore proof valid against current Merkle root
  - Nullifier not already used
  - Ballot validity proof valid
  All pass → ballot accepted and recorded
        │
        ▼
Contract updates homomorphic tally:
  tally[0] = tally[0] * enc(0)
  tally[1] = tally[1] * enc(1)
  tally[2] = tally[2] * enc(0)

Nullifier stored on-chain → voter cannot vote again
TS Indexer detects VoteCast event → updates Merkle snapshot
```

### 6.9 Tally — Partial Decryptions

Final encrypted tally C is on-chain:
  C = [ enc(totalAlice), enc(totalBob), enc(totalCarol) ]

Any k of n guardians must participate. Each independently:

```
Guardian logs in (wallet auth)
        │
        ▼
Browser fetches encrypted tally C from chain
        │
        ▼
Guardian loads their encrypted secret share file (downloaded during key ceremony)
Browser decrypts secret share using guardian's Ethereum private key
        │
        ▼
Browser (Rust WASM) computes partial decryption:
  Di[j] = C[j]^si mod p   (for each candidate position j)
        │
        ▼
Browser generates Chaum-Pedersen ZK proof:
  proves: Di was computed using the same si as the registered Pi
  without revealing: si
        │
        ▼
Guardian submits:
  { partialDecryption: Di[], zkProof }
Secret share used in-memory only — never uploaded
        │
        ▼
Spring Boot collects submissions
Once k guardians have submitted:
  Rust sidecar combines:
    D[j] = D1[j] * D2[j] * D3[j]   (for each candidate j, k=3)
    finalTally[j] = discrete_log(D[j])
    (feasible because vote counts are small integers)
        │
        ▼
Final tally computed:
  [ Alice: 1201, Bob: 4823, Carol: 892 ]
```

### 6.10 Result Publication and Audit

```
Rust sidecar produces full audit package:
  {
    electionId,
    electionPublicKey K,
    guardianPublicShares[],
    guardianKeyShareProofs[],
    encryptedTally,
    partialDecryptions[],
    partialDecryptionProofs[],
    finalTally,
    verificationInstructions
  }
        │
        ▼
Audit package uploaded to IPFS
IPFS returns auditCID
        │
        ▼
TS TX Service submits:
  Election.finalizeResult(finalTally, auditCID)
        │
        ▼
Result and audit CID stored on-chain permanently
Status: FINALIZED

Anyone can independently:
  - read result from chain
  - fetch full audit package from IPFS using auditCID
  - verify every ZK proof
  - confirm tally matches on-chain encrypted ballots
  - verify no ballot was added, removed, or modified
```

## 7. Election Status Lifecycle

```
DRAFT
  → (2 of 3 admin approvals) →
APPROVED
  → (all guardians submit key shares) →
KEY_CEREMONY_COMPLETE
  → (election deployed on-chain) →
PENDING
  → (voters registered, authority opens election) →
ACTIVE
  → (authority closes election) →
ENDED
  → (k guardians submit partial decryptions) →
TALLY
  → (result finalized, audit published) →
FINALIZED
```

## 8. What the Platform Cannot Do

Trustlessness guarantees — cryptographically impossible even for the operator:

| Action | Why impossible |
|---|---|
| Read any individual vote | Encrypted client-side with ElGamal — server never sees plaintext |
| Forge a vote | Requires valid Semaphore proof — needs voter's private key |
| Double count a vote | Nullifiers enforced on-chain |
| Decrypt the tally alone | Requires k guardian secret shares — platform holds none |
| Deploy an election unilaterally | Requires 2 of 3 platform Safe signers on-chain |
| Register voters unilaterally | Requires 2 of election authority Safe signers on-chain |
| Modify election metadata after creation | On-chain CID is immutable |
| Suppress audit evidence | Audit CID on-chain — anyone can fetch from IPFS and verify |
| Censor votes silently | Voter holds signed receipt — absence provable against on-chain data |
